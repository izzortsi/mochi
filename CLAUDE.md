# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running

```bash
./start.sh                    # backend on :4000, frontend on :3000
./start.sh 4001 3001          # override ports
```

Manually:

```bash
# backend (uv manages the venv + deps from pyproject.toml)
cd backend && uv run uvicorn app.main:app --host 0.0.0.0 --port 4000

# frontend (custom script picks next free port starting from $PORT)
cd frontend && npm install && npm run dev
```

No test suite, linter, or build step is wired up beyond `next build` / `next lint`.

## Architecture

Two processes, one user:

- **`backend/app/`** — FastAPI app (`main.py`) mounting routers from `app/routers/` and a WebSocket at `/ws`. Persistence is plain JSON files under `backend/data/` guarded by a single `threading.Lock` in `store.py`. There is no database.
- **`frontend/src/`** — Next.js 14 App Router. `next.config.js` proxies `/api/*` → `localhost:4000/api/*`, so the browser only sees `:3000`. WebSocket connects directly to `:4000/ws` via `lib/ws.ts`.

### Wire format convention

The API uses **kebab-case JSON keys** (e.g. `card-uid`, `best-streak`, `course-id`, `day-tiers`). The frontend runs every response through `camelizeKeys` in `lib/api.ts` before typing it, and sends kebab-case keys in request bodies. `camelizeKeys` deliberately **skips** keys matching `/^c\d+-d\d+-/` (card UIDs) and `/^\d+\.\d+$/` (day-tier keys) so they stay intact as map keys. Don't remove this exemption.

Backend handlers frequently accept both camelCase and kebab-case in request bodies (see `routers/ws.py` `args.get("courseId") or args.get("course-id")`) — when adding new endpoints, follow the same dual-read pattern for bodies and emit kebab-case in responses.

### Card UID format

Every card has a synthetic id: `c{courseId}-d{dayId}-{tier}-{taskIndex}` (e.g. `c1-d3-silver-0`). Generated by `gamification.card_uid_for`. Tier is `bronze`/`silver`/`gold`. Several routers parse this string directly to recover course/day/tier — don't change the format.

Day-tier map keys use `{courseId}.{dayId}` (e.g. `"1.3"`).

### WebSocket tool dispatch (`routers/ws.py`)

The WS channel is a JSON request/response RPC, not streaming. Messages: `{requestId, call, args}` → `{ok, result, requestId}` or `{ok: false, error, requestId}`. Tools registered in the `handlers` dict in `dispatch()` — add new ones there. This is the surface the LLM tutor calls for live state reads/mutations (next-up, concept-mastery, mark-task-complete, append-chat, etc.).

### LLM providers

Two providers, selected by `provider` field in request body:

- **`zai`** — OpenAI-compatible HTTP. API key comes from the browser (`localStorage`), passed per-request as `api-key`. Never stored server-side.
- **`anthropic-oauth`** — uses the `anthropic-oauth` package (sourced from `https://github.com/izzortsi/anthropic-oauth` via `[tool.uv.sources]`). Requires a one-time `anthropic-oauth auth` in a terminal to seed tokens; the server reads them via `OAuthManager`. If no tokens exist, `llm_anthropic._get_client` raises `RuntimeError` which surfaces as HTTP 503.

System prompts for PDF import live in `services/llm.py` (`SYSTEM_PROMPT_NEW`, `SYSTEM_PROMPT_EXTEND`). They enforce singular kebab-case concept IDs and reuse of existing concepts across courses — don't drop the "existing concepts" block when editing.

### Concept aliases

`data/aliases.json` maps alternate concept IDs to a canonical form. `routers/courses.resolve_concept_id` is called at nearly every boundary (import, tag-card, concept map build, prereq lookup) to normalize raw IDs. When adding code that reads or writes `card.concepts` or `course.prereqs`, resolve through aliases — mixing raw and canonical IDs silently breaks the concept map.

### Data files

All live under `backend/data/` and are now git-tracked so sessions resume across machines:

- `courses.json` — list of `CourseDef` (phases, days, cards, concepts, prereqs)
- `progress.json` — XP, streak, `completed_tasks`, `day_tiers`, attempts, generated_tasks
- `pet.json` — Tamagotchi state (stage, health, happiness, art). Health/happiness decay in `routers/pet._decay_pet` on every GET.
- `notes.json` — Zettelkasten notes with bidirectional `related` links
- `aliases.json` — concept ID canonicalization
- `chat.json` — per-course chat history (created on first write)
- `ocr/` — cached OCR output keyed by PDF hash

PDFs uploaded to `backend/pdfs/` (gitignored). `SP_*` env vars override settings in `app/config.py` (`SP_OLLAMA_URL`, `SP_OLLAMA_MODEL`, `SP_DATA_DIR`, etc.).

### Gamification coupling

`routers/progress.complete_card` does three things in order: updates XP, recomputes the day tier, then calls `routers.pet.feed_pet`. A gold day triggers an extra `feed_pet("gold_bonus", 0)`. If you add a new completion path (e.g. via WS `mark-task-complete`), mirror this sequence or the pet will silently starve.

## Conventions

- Python: `from __future__ import annotations` at the top of every module; Pydantic v2 models in `app/models.py`; no ORM.
- Frontend: path alias `@/*` → `src/*`; TailwindCSS; KaTeX for math; components are client components (`"use client"`) unless trivially static.
- Ports: backend defaults to 4000, frontend to 3000. The frontend dev script (`scripts/next-dev.js`) auto-increments if `$PORT` is busy; backend refuses to start on a busy port.
